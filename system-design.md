# SyncOrbit — System Design & Architecture

SyncOrbit is a real-time collaborative text editing platform engineered for scale, resilience, and ultra-low latency. It is built around **Yjs** (a Conflict-Free Replicated Data Type framework), enabling seamless peer-to-peer conflict resolution.

This document outlines the core architecture, data flow pipeline, state mapping, and scaling guarantees.

---

## 1. High-Level Architecture

The platform follows a three-tier architecture with a horizontally scalable WebSocket gateway layer.

### Tech Stack
- **Frontend**: Next.js (React), Tiptap editor, Yjs, Tailwind CSS
- **WebSocket Gateway**: Node.js `ws` server, `yjs`
- **Volatile/PubSub Layer**: Redis (ioredis)
- **Persistence Layer**: MongoDB (Mongoose)

## 2. The Real-Time Data Pipeline

Every CRDT update generated by a user's browser travels through a strictly ordered pipeline. This ensures consistency across instances while maintaining sub-30ms local latency.

### Pipeline Stages

1. **Client Generation**: A user types a character. Tiptap modifies its internal Y.Doc. The `update` event yields a binary differential update.
2. **WebSocket Ingest**: The binary payload (`Uint8Array`) arrives at the Node Gateway.
3. **Local Application (Server Y.Doc)**: The Gateway applies the binary update to its **in-memory server Y.Doc replica**. It then immediately forwards the update to all other connected clients in the same room on the same node.
4. **Redis Distribution**: The Gateway encapsulates the payload with its unique `SERVER_ID` and publishes it to the Redis channel `doc:{docId}`.
5. **Cross-Node Subscriber**: Other Node instances subscribed to `doc:{docId}` receive the payload. They verify `originId !== SERVER_ID` (preventing self-echo), apply it to their local Y.Doc, and broadcast to their respective connected clients.
6. **Snapshot Persistence**: Upon applying an update, a counter triggers (`TRACK_UPDATE`). Periodically (e.g., every 50 updates or 5 minutes), the Gateway encodes the entire Y.Doc state and saves a binary snapshot to MongoDB.

## 3. The Two-Dimensional State Guard

To prevent race conditions when horizontally scaling (e.g., two clients hit two different Node servers for an unloaded document at the exact same millisecond), the system utilizes `documentManager.js`.

The manager controls the lifecycle linearly via a **Two-Map Guard**:
- **`loadingDocs: Map<String, Promise<Room>>`** 
- **`rooms: Map<String, Room>`**

### The `getOrCreateRoom` Flow
1. Check `rooms`. If loaded, return instantly.
2. Check `loadingDocs` (Dedup Guard). If another request is currently loading this document, **await the same promise**.
3. Fallback: Create load Promise, set in `loadingDocs`, fetch from MongoDB, hydrate Y.Doc, move to `rooms`.

*Guarantee*: Exactly **one** database fetch operation per docId per server instance.

## 4. Awareness vs. CRDT Data

Data passing over the WebSocket is prefixed by a single **Message Type Byte**:

| Byte | Type | Description | Handling |
|------|------|-------------|----------|
| `0x00` | CRDT Document Update | Actual text changes | Applied to Y.Doc, published to Redis, persisted to MongoDB. |
| `0x01` | Awareness Update | Cursor positions, selections, presence | Ephemeral. Forwarded to local room clients only. No Redis. No DB. |

By separating these streams, the server avoids polluting the persistence layer or exhausting Redis Pub/Sub bandwidth with 60fps cursor movements.

## 5. Performance Optimizations

- **Awareness Throttling**: The frontend uses a 50ms trailing-edge throttle. Rapid cursor movements are collapsed, ensuring the network is not flooded, yet the final resting cursor position is always transmitted.
- **Batched Presence Debouncing**: Connection/disconnection events trigger a `broadcastPresence` function that is internally debounced (100ms per room). A burst of 10 users joining yields only 1 JSON broadcast to the room.
- **Optimized Broadcast Loops**: Utilizing pre-allocated `Buffer.concat` and native `for..of` loop constructs to minimize garbage collection overhead per broadcast tick.
- **Graceful Eviction (GC)**: Idle rooms (no connected clients for 10 minutes) trigger `evictRoom()`, which saves a final MongoDB snapshot and removes the instances from memory.
